1. Purpose
This document is the authoritative task specification for UKCP Sprint 4. It defines, in order of execution, every task that Claude Code (Dex) must deliver. Each task includes the files to be created or modified, the technical requirements, and the acceptance conditions that must be met before the task is considered complete.

Nothing in this document is implemented until Phil has confirmed scope. Nothing is accepted until Phil has tested it. Ali owns this document and updates it as the sprint progresses.

2. Sprint Goal
Sprint 4 resolves two issues discovered during Sprint 3 runtime review:

First, the visual structure requires polish. The header, body columns, and footer need a constrained content width (95%, centred) and subtle row separators so each structural section is visually distinct. The backgrounds remain full-width; only the content zones shrink.

Second, the hierarchy navigation is completely broken. The Sprint 3 design deprecated the two-panel HeaderNavigation component and replaced it with a stepper walker in Row 3 — but the stepper only handles back-navigation. Forward navigation (selecting region, county, area, constituency) has no UI. The panel1, panel2, and select values from useNavigation are destructured in Locations.jsx but never passed to any component. Sprint 4 restores forward navigation by creating a LocationNav component that lives in the mid pane. The mid pane splits into two vertical sections: LocationNav on top (always visible when walker is open), PlacesCard below (appears only once constituency is reached, which is the common key between the hierarchy dataset and the places dataset).

At sprint end, the header must be visually polished at 95% content width, hierarchy navigation must work end-to-end in the mid pane, and PlacesCard must appear filtered by county once the user navigates to constituency level.

4. Baseline State
Sprint 3 delivered the four-row header scaffold with a clean build (0 errors, 0 warnings, 6874 modules). The following issues were identified at runtime review:

Issue
Description
Header full-width
Row content extends to screen edges. Should be 95% width, centred. Row backgrounds stay full-width.
No row distinction
Header rows with identical backgrounds run together visually. Thin separator needed between each row.
No section gaps
Body columns butt directly against the header bottom and footer top. A small gap improves legibility.
Footer full-width
Footer content should match the constrained header width.
Navigation broken
panel1, panel2, and select from useNavigation are unused. HeaderNavigation was deprecated but no forward navigation was provided. The walker stepper (Row 3) handles back-nav only. The mid pane shows PlacesCard but it can never load because county is never reachable via the UI.

4.1 Current Mid Pane Wiring
Locations.jsx calls useNavigation and receives { path, panel1, panel2, select, goTo, reset }. Only path, goTo, and reset are currently used. panel1, panel2, and select are destructured but discarded.

4.2 usePlacesFilter Behaviour
usePlacesFilter(places, path) derives county = path.find(p => p.level === 'county')?.value. countyActive is true when county is not null. filterPlaces filters place.csv rows by matching place.County to the county in the path. Because the user cannot navigate to county level (no forward nav UI), countyActive is always false at runtime.

4.3 Constituency as Common Key
The hierarchy dataset (newplace.csv) and the places dataset (place.csv) share constituency as the common key. The places filter is based on county (the parent of constituency in the path), meaning PlacesCard shows all places in the county containing the selected constituency. PlacesCard should not appear until the user has navigated to constituency level — that is the semantic point at which both datasets are in scope.

6. Standards (applies to all tasks)
These are binding. Any file that fails these standards is rejected without further review.

Standard
Requirement
JSDoc

No style={} props. Mantine className with CSS module, or Mantine layout props only.
