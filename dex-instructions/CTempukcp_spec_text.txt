1.  Purpose
This document is the authoritative task specification for UKCP Sprint 1. It defines, in order of execution, every task that Claude Code (Dex) must deliver. Each task includes the files to be created, the data and logic requirements, and the acceptance conditions that must be met before the task is considered complete.

Nothing in this document is implemented until Phil has confirmed scope. Nothing is accepted until Phil has tested it. Ali owns this document and updates it as the sprint progresses.

2.  Sprint Goal
Deliver a working Locations feature on the clean UKCP scaffold. At sprint end, a user must be able to navigate the full UK administrative hierarchy, see physical places filtered to their navigation context, and select a location that is persisted to sessionStorage.

The application must build cleanly with Vite 5 and serve correctly via Express on port 3000.

4.  Data Notes (read before writing any data code)
These notes apply to all tasks. Dex must read and apply them.

4.1  newplace.csv
72,467 rows. Dual-purpose file. Two record types distinguished by the type column:

type value
Count
Purpose
LOC
61,587
Locale names — drives the hierarchy navigation tree. These are the navigable entries.
WD
10,880
Ward records — the nameid on these rows matches wardid in wards.csv.

Columns: nameid, name, type, country, region, county, area, constituency, lat, long
Hierarchy built from LOC rows only. Filter on type === 'LOC' before building the tree.

4.2  place.csv
43,338 rows. Physical settlements: City (69), Town (1,566), Village (17,598), Hamlet (24,105).
Columns: PlaceName, Lat, Lng, Type, County, AdCounty, District, UniAuth, Country
Joined to navigation context via County. This is a soft string match — normalise (trim, lowercase) before comparing. No IPN codes on this file.

4.3  wards.csv
10,861 rows. Ward identity data. Columns: country, region, county, area, constituency, ward, wardid.
wardid is an IPN code that matches nameid on WD-type rows in newplace.csv. Not used in Sprint 1 but must not be confused with the LOC data.

4.4  Data Quality
The data was assembled from multiple state sources (ONS, OS Names) and has known vagaries: encoding anomalies, naming convention differences between nations, and soft-join imprecision between files. Dex must handle this defensively:
Wrap all CSV parsing in try/catch
Skip and log any row that cannot be placed in the hierarchy — do not throw
Normalise all string comparisons: trim whitespace, lowercase
Never assume a field is populated — guard for empty or null values

T2
Data Loader Hook
Owner: Dex | Reviewer: Ali

Goal
Load and cache both CSV datasets. newplace.csv is parsed and built into a navigable hierarchy object stored in localStorage. place.csv is parsed and stored as a flat array in localStorage. Both are version-keyed.
Branch
sprint1/T2-data-loader
Deliverables
All files listed below, complete and JSDoc-compliant. No snippets.
Dependency
T1 accepted.

Files to Create
File
Purpose
src/hooks/useLocationData.js
Primary data hook. Loads both CSVs, builds hierarchy, writes to localStorage, exposes loading state and error state.
src/utils/buildHierarchy.js
Pure utility function. Takes parsed newplace.csv LOC rows, returns nested hierarchy object. Separated for testability.
src/utils/cacheKeys.js
Exports CACHE_KEY_WARDS and CACHE_KEY_PLACES constants. Version-keyed (e.g. UKCP_WARDS_v1).

Hierarchy Structure
buildHierarchy.js must produce the following nested object from LOC-type rows only:

Level
Structure
Root
{ [country]: { regions: { ... } } }
Region
{ [region]: { counties: { ... } } }
County
{ [county]: { areas: { ... } } }
Area
{ [area]: { constituencies: { ... } } }
Constituency
{ [constituency]: { names: [ { nameid, name, lat, long } ] } }

Names (LOC entries) hang off the constituency level as an array, not a further nested object. Each entry preserves nameid, name, lat, and long for future use.

Caching Rules
Check localStorage for version-keyed cache on mount. If present, use it. Do not re-parse.
If absent or key mismatch, parse CSV, build hierarchy, write to localStorage.
place.csv stored as flat JSON array. No hierarchy required.
Both writes wrapped in try/catch — localStorage quota errors must be caught and surfaced via error state, not thrown.
Hook exposes: { hierarchy, places, loading, error }.

Hierarchy Navigation & Crumb Trail
Owner: Dex | Reviewer: Ali

Goal
Build the two-panel navigation header and crumb trail. The user can drill from region to constituency, return to any prior level via the crumb trail, and the selected path is persisted to sessionStorage.
Branch
sprint1/T3-navigation
Deliverables
All files listed below, complete and JSDoc-compliant. No snippets.
Dependency
T2 accepted.

Files to Create
File
Purpose
src/components/SiteHeader.jsx
Top-level header wrapper. Contains logo area, primary nav icons, and renders HeaderNavigation below it.
src/components/HeaderNavigation.jsx
Two-panel navigation component. Panel 1: regions list. Panel 2: dynamic child list (counties, areas, constituencies). Controlled by useNavigation.
src/components/CrumbTrail.jsx
Renders the current selection path as a horizontal trail of clickable labels. Clicking any crumb returns navigation to that level.
src/hooks/useNavigation.js
Navigation state hook. Manages the selection path array, current panel content, and sessionStorage persistence. Exposes select(level, value), goTo(index), reset(), and path.
src/utils/getChildren.js
Pure utility. Given the hierarchy object and a selection path, returns the child array for the next navigation level.

Navigation Behaviour
Places icon in SiteHeader activates the navigation panels.
Panel 1 always shows the list of regions (top-level keys of the hierarchy, filtered to exclude Scotland until open decision 4 is resolved — see note below).
Selecting a region populates Panel 2 with counties in that region. Panel 1 locks to show selected region.
Each subsequent selection updates Panel 2 with the next level. Path: region → county → area → constituency.
At constituency level, the panel shows the LOC names (locale names) within that constituency.
CrumbTrail renders below the panels. Each crumb is clickable. Clicking a crumb resets the path to that index and re-renders Panel 2 from that level.
Full path is written to sessionStorage on every selection. Restored on page refresh.

Note: Scotland / Wales / NI Hierarchy (Open Decision 4)
Open Decision 4 is unresolved. The prior codebase used All-Scottish Regions as a single flat region entry, which is a known fudge. For Sprint 1, exclude Scotland, Wales, and Northern Ireland from Panel 1 and build English hierarchy only. This is explicitly temporary. A note must be added to the code as a TODO comment referencing Open Decision 4.

T4
Places Browser
Owner: Dex | Reviewer: Ali

Goal
Render the physical places browser below the navigation panels. Cities, towns, villages, and hamlets are filtered to the current navigation context (by County match). Selecting a place name triggers the location selection banner.
Branch
sprint1/T4-places-browser
Deliverables
All files listed below, complete and JSDoc-compliant. No snippets.
Dependency
T3 accepted.

Files to Create
File
Purpose
src/components/PlacesCard.jsx
Renders the filtered list of physical places. Grouped by Type (City, Town, Village, Hamlet). Each place name is clickable.
src/hooks/usePlacesFilter.js
Filters the cached places array against the current navigation context. Returns filtered and grouped results.
src/components/SelectionBanner.jsx
Banner shown when a place name is clicked. Displays the selected place name and a Select button. Clicking Select writes the location to sessionStorage and emits a confirmed selection state.
src/utils/filterPlaces.js
Pure utility. Takes the flat places array and a county string, returns filtered places normalised and sorted by Type then PlaceName.

Filtering Logic
Filter field: place.csv County column matched against the county value in the current navigation path.
Match is case-insensitive, trimmed. Not exact — use includes() or normalised equality. Source data has known inconsistencies.
If no county is selected in the navigation path yet, PlacesCard renders empty with a prompt: "Navigate to a county to see places."
Results grouped: Cities first, then Towns, Villages, Hamlets. Within each group, alphabetical.

Selection Flow
Clicking a PlaceName in PlacesCard sets it as the pending selection and shows SelectionBanner.
SelectionBanner displays: the place name, its type, and the current crumb trail context.
Clicking Select in SelectionBanner writes to sessionStorage: { placeName, type, county, crumbTrail }.
On confirmed selection, the header banner label updates to the selected place name.
Clicking anywhere outside the banner (or a dismiss control) cancels the pending selection without committing.

Acceptance Conditions
AC
Acceptance Condition
AC-1
With a county selected in navigation, PlacesCard renders places filtered to that county, grouped and alphabetically sorted.
AC-2
With no county selected, PlacesCard shows the "Navigate to a county to see places" prompt.
AC-3
Clicking a place name shows SelectionBanner with the correct place name, type, and crumb context.
AC-4
Clicking Select writes the correct object to sessionStorage. No page reload occurs.
AC-5
After selection, the header banner reflects the selected place name.
AC-6
Dismissing the banner without clicking Select does not write to sessionStorage.

6.  Standards (applies to all tasks)
These are binding. Any file that fails these standards is rejected without further review.

Standard
Requirement
JSDoc
Every file has a file-level JSDoc block. Every exported function has a JSDoc block with @param and @returns.
File naming
Components: PascalCase.jsx | Hooks: camelCase with use prefix | Utils: camelCase | Constants: UPPER_SNAKE_CASE
No TypeScript
All files are .js or .jsx. No .ts or .tsx.
No inline styles
No style={} props. Use Mantine className or CSS modules only.
No sx prop
Mantine v7: sx still exists but is deprecated-adjacent. Do not use it.
One responsibility
Each file does one thing. If a component fetches, transforms, and renders — split it.
Props only
Components communicate via props. No component reaches into another's state.
Complete files
Dex delivers complete files only. No snippets, no partial updates, no "add this block" instructions.
